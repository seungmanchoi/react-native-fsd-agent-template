# Google Play 프로덕션 액세스 신청 (Production Access Application)

Google Play **개인(personal) 개발자 계정**으로 만든 신규 앱을 프로덕션(공개) 트랙에 올리려면, 먼저 **"프로덕션 액세스 신청"** 양식을 제출하고 Google 심사를 통과해야 한다. 이 문서는 그 신청 양식의 전체 질문과 **답변 작성 원칙**을 정의한다. Phase 7 배포(특히 Android, 개인 계정)에서 `api-integrator`의 "Play Console 프로덕션 액세스 신청 자동화" 능력이 이 문서를 정본으로 참고한다.

> 관련: `deploy-build-troubleshooting.md`(빌드/제출), `~/works/store-deploy-plugin/skills/store-forms/SKILL.md`(스토어 폼 일반).

---

## 1. 적용 조건 / 게이트

개인 계정은 아래 3가지를 **모두** 충족해야 "프로덕션 신청" 버튼이 활성화된다 (앱 대시보드 → 프로덕션 카드에 체크 표시):

1. 비공개 테스트(closed testing) 버전 게시
2. **12명 이상**의 테스터가 비공개 테스트 참여 opt-in
3. **12명 이상** 테스터로 **14일 이상** 비공개 테스트 실행

세 조건이 ✅✅✅ 되기 전엔 `eas.json`의 track을 production으로 바꿔도 공개 출시 불가. 법인(조직) 계정은 이 게이트가 없을 수 있다.

- **신청 위치**: 앱 대시보드 → "프로덕션" 카드 → **"프로덕션 신청"** 버튼 → 3단계 다이얼로그
- **심사**: 제출 후 Google 검토. **보통 7일 이내**(더 걸릴 수 있음). 결과는 **계정 소유자 이메일**로 통보
- **승인 전 상태**: 프로덕션 트랙 "비활성" → 신청 외엔 할 일 없음(대기)

## 2. 신청 양식 — 전체 질문 (3단계)

각 텍스트 필드는 **300자 제한**(글자수 카운터, 한국어 입력 가능). 라디오는 단일 선택.

### Step 1 — 비공개 테스트 정보
| # | 질문 | 입력 형식 |
|---|------|----------|
| 1 | 비공개 테스트에 참여할 사용자를 어떻게 모집하셨나요? (친구·가족 / 유료 테스트 업체 등) | 텍스트 300자 |
| 2 | 게임/앱 테스터를 얼마나 쉽게 모집했나요? | 라디오: 매우 어려웠음 / 어려웠음 / 보통이었음 / 쉬웠음 / 매우 쉬웠음 |
| 3 | 비공개 테스트 중 테스터의 참여도 (모든 기능 사용했는지, 실제 사용자 플레이와 일치하는지, 다르면 어떤 차이) | 텍스트 300자 |
| 4 | 테스터로부터 받은 의견 요약 + 의견 수집 방법 | 텍스트 300자 |

### Step 2 — 게임/앱 정보 (Google Play 미표시, 자격요건 영향 없음)
| # | 질문 | 입력 형식 |
|---|------|----------|
| 5 | 주요 대상은 누구인가요? | 텍스트 300자 |
| 6 | 앱/게임을 돋보이게 하는 요소 | 텍스트 300자 |
| 7 | 첫해 예상 설치 수 | 라디오: 0~1만 / 1만~10만 / 10만~1백만 / 1백만 이상 / 모름 |

### Step 3 — 프로덕션 준비
| # | 질문 | 입력 형식 |
|---|------|----------|
| 8 | 비공개 테스트에서 알게 된 내용을 바탕으로 어떻게 변경했나요? | 텍스트 300자 |
| 9 | 프로덕션용으로 준비되었다고 어떻게 판단했나요? | 텍스트 300자 |

마지막 단계의 **"적용"** 버튼이 최종 제출이다.

## 3. 답변 작성 원칙 (CRITICAL)

1. **사실 기반 — 절대 날조 금지.** 답변은 반드시 개발자(사용자)에게 실제 사실을 먼저 물어 그 사실에 충실하게 작성한다. 테스터 수·피드백·지표를 지어내지 않는다. Google 심사에 직접 영향을 주며, 허위 진술은 거부·정책 위반 사유가 된다.
2. **성실하고 구체적으로.** 한 문장짜리 답변·예시 문구 복붙은 지양. 실제 정황(누구에게 어떻게 부탁했고, 어떤 채널로 피드백을 받았고, 무엇을 고쳤는지)을 구체적으로 풀어 쓴다.
3. **담백한 톤, 과장 금지.** 개인 인디 앱 맥락에 맞게 솔직하게. "친구·가족에게 부탁", "메신저로 피드백 수집", "크래시/ANR 콘솔 데이터로 수정" 같은 현실적 서술이 신뢰도를 높인다.
4. **300자 이내**로 맞추되 내용이 빈약하지 않게. 핵심 사실 → 부연 1~2문장 구조.
5. **설치 예상은 정직하게.** 개인 인디 게임/앱은 보통 **0~1만**. 근거 없이 큰 수를 고르지 않는다.
6. **안정성·테스터 동의를 준비 근거로.** "크래시·ANR 등 알려진 오류 해결 + 테스터들이 출시 동의"는 Step 3의 설득력 있는 근거다.
7. **최종 제출 전 사용자 확인.** 제출은 Google에 보내는 형식적·되돌리기 어려운 요청이므로, 작성한 답변을 사용자에게 보여주고 "적용" 직전에 확인을 받는다(사용자가 명시적으로 제출을 지시한 경우 제외).

## 4. 작성 전 사용자에게 물어볼 facts (체크리스트)

자동화 전에 아래를 수집한다(모르면 추정/날조 대신 사용자에게 질문):
- 테스터를 **어떻게** 모았나 (친구/가족/지인/유료 업체)
- 모집이 얼마나 **쉬웠나** (5단계 중)
- 테스터가 앱을 **어떻게 사용**했나 (상시 vs 가끔, 실제 사용자와 차이)
- 받은 **피드백 내용**과 **수집 채널** (카카오톡·메신저·설문 등)
- **주요 대상** 사용자
- 앱의 **차별점**
- 첫해 **설치 예상** (보통 0~1만)
- 테스트로 **무엇을 바꿨나** (크래시/ANR 수정, UX 개선)
- **준비됐다고 판단한 근거** (오류 해결, 테스터 동의)

## 5. Worked example — Block Aurora (2026-06-22 제출, 캐주얼 블록 퍼즐)

> ⚠️ 그대로 복붙하지 말 것. 각 앱의 실제 사실로 다시 쓴다. 톤/구체성/길이의 기준점으로만 사용.

- **(1) 모집 방법**: "가족과 친구들에게 직접 부탁하여 테스터를 모집했습니다. 유료 테스트 제공업체는 이용하지 않았으며, 평소 모바일 게임을 즐기는 지인들에게 개별적으로 연락해 비공개 테스트 참여를 요청하고 설치 및 옵트인 과정을 직접 안내했습니다."
- **(2) 모집 난이도**: 매우 어려웠음
- **(3) 참여도**: "상시 플레이하는 유형이 아니라 무료한 시간에 잠깐씩 즐기는 캐주얼 퍼즐이라, 테스터들도 실제 사용자와 동일하게 필요할 때만 플레이했습니다. 다만 테스터 관점에서 평소보다 더 꼼꼼히 관찰하며 다양한 기능을 사용해 주었고, 실제 사용자의 플레이 방식과 크게 다르지 않을 것으로 예상합니다."
- **(4) 피드백+수집**: "피드백은 주로 카카오톡 등 메신저로 수집했습니다. 조작감과 난이도, 일부 UI의 불편한 점에 대한 의견과 함께, 비정상 종료(크래시)나 ANR 같은 안정성 문제도 보고받았습니다. 받은 의견을 정리해 우선순위를 정한 뒤 게임 수정에 반영했습니다."
- **(5) 주요 대상**: "무료한 시간에 가볍게 즐길 거리를 찾는 캐주얼 게임 사용자입니다. 복잡한 학습 없이 짧은 시간에 몰입할 수 있는 블록 퍼즐을 선호하는, 특정 연령에 치우치지 않은 폭넓은 연령대의 일반 사용자를 주요 대상으로 합니다."
- **(6) 차별점**: "누구나 직관적으로 이해할 수 있는 간단한 블록 퍼즐 규칙에, 네온 오로라 콘셉트의 비주얼과 라인이 사라질 때의 만족스러운 연출을 더했습니다. 짧은 플레이로도 부담 없이 무료함을 달랠 수 있는 가벼운 몰입감이 강점입니다."
- **(7) 첫해 설치**: 0~1만
- **(8) 변경 사항**: "Google Play Console에 보고된 비정상 종료(크래시)와 ANR 데이터를 분석해 안정성 문제를 수정했습니다. 또한 테스터들이 메신저로 전달한 조작감과 UI 관련 불편 사항을 반영해 사용성을 개선했고, 테스트 기간 동안 사용성을 지속적으로 모니터링하며 보완했습니다."
- **(9) 준비 판단**: "주요 크래시와 ANR 등 알려진 오류가 해결되어 안정적으로 동작하는 것을 확인했습니다. 또한 14일 이상 비공개 테스트에 참여한 테스터들이 직접 플레이한 뒤 프로덕션으로 출시해도 좋겠다는 의견을 모아주어, 출시 준비가 되었다고 판단했습니다."

## 6. 승인 후 출시 (참고)

프로덕션 액세스가 승인되면:
- **권장**: 콘솔에서 비공개 테스트(alpha) 최신 버전을 **"버전 승급" → 프로덕션**. 재빌드 없이 이미 테스트된 동일 AAB가 출시됨.
- 또는 `eas.json`의 `submit.production.android.track`을 `"production"`으로 바꾸고 새 빌드를 제출.
- 출시 전 프로덕션 트랙 **국가/지역**과 **출시 노트**(Android changelog 500byte 제한) 확인.

> `eas.json` android `releaseStatus: "completed"`가 아니면 제출이 draft로만 올라가 테스터/사용자에게 안 나간다. (raw Play API promote로 깨거나 releaseStatus="completed" 사용 — `deploy-build-troubleshooting.md` 참고.)
